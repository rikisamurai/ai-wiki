## 一场被时间戳打穿的缓存事故

周一早上，团队收到一条账单告警。

一个代码审查 Agent 的调用量没有明显上涨，模型没有切换，用户数甚至比上周略少。但 API 成本一夜之间翻了几倍，P90 延迟也从 2 秒左右涨到 10 秒以上。

排查日志时，大家一开始盯着模型输出、工具调用次数、上下文长度，以为是 Agent 走了更多步骤。最后，问题藏在 Prompt 的最前面：current_time: 2026-04-21 09:03:12。

这行时间戳每次请求都会变化，而它被放在系统提示词、工具定义、代码库说明之前。

于是模型每次看到的输入前缀都不一样。那 20,000 tokens 的系统提示词和工具 Schema，本来应该只计算一次、反复复用，现在每一步工具调用都要重新 Prefill 一遍。

第 12 次工具调用，重算一遍。第 13 次工具调用，再重算一遍。第 40 次工具调用，还是重算一遍。

没有报错，没有异常，只是钱在烧，用户在等。

Prefix Cache 的价值就在这里：它不是一个“锦上添花”的性能优化，而是长链路 Agent 能不能跑得起、跑得快的基础设施。



## 一、什么是 Prefix Cache

### 1.1 从 KV Cache 说起

每一个基于 Transformer 架构的大语言模型，在推理过程中都需要计算注意力机制（Self-Attention）。对序列中的每个 Token，模型会在每一层生成对应的 **Key (K)** 和 **Value (V)** 张量。这些张量代表该 Token 在当前上下文中的语义表示。

在单次推理中，**KV Cache** 是标配优化：模型生成第 N 个输出 Token 时，不需要重新计算之前所有 Token 的 K/V 值，而是直接复用已缓存的结果。没有 KV Cache，生成第 500 个 Token 就需要对前 499 个 Token 全部重新做前向传播，代价极高。

把它想象成写作：你每写一个新字，不需要把前面所有字重新理解一遍——你的大脑已经把前文的语境"缓存"起来了。KV Cache 做的，正是同样的事。

### 1.2 Prefix Cache：跨请求的 KV 共享

KV Cache 解决的是**单次请求内部**的重复计算问题。**Prefix Cache（前缀缓存）** 则更进一步，把复用范围扩展到了**多个独立请求之间**。

类比一下：KV Cache 像是一节课里老师不用反复重读黑板上的内容；而 Prefix Cache，则是第二节课、第三节课来的同学，也能直接继承黑板上的板书，不用让老师从头再写一遍 📋

其核心思路是：当多个请求共享相同的 Token 序列前缀（如相同的系统提示词、工具定义、参考文档），就没有必要对每个请求都从头计算一遍这段前缀的 K/V 张量。只需计算一次，后续所有请求直接读取缓存。

### 1.3 KV Cache vs Prefix Cache：核心区别

两者是互补的搭档，不是竞争关系 🤝

|维度|KV Cache|Prefix Cache|
|---|---|---|
|**作用范围**|单次请求内部|跨多个独立请求|
|**优化阶段**|Decode（生成）阶段|Prefill（预填充）阶段|
|**是否默认开启**|是，始终开启|需要条件满足（相同前缀）|
|**内存共享**|每个请求独占|多请求共享同一物理内存块|
|**节省类型**|避免解码时的重复计算|避免预填充阶段的重复计算|

KV Cache 是基础机制，Prefix Cache 是其跨请求的延伸。在生产 Agent 系统中，两者同时运作，缺一不可。

### 1.4 工作原理：三步流程

理解 Prefix Cache 的运作，只需要记住三个阶段：

**⚙️ Prefill 阶段**：模型对输入 Prompt 做完整前向传播，生成所有 Token 的 KV 张量。这是计算最密集的阶段，耗时与 Prompt 长度正相关——前缀越长，这里越痛。

**🖊️ Decode 阶段**：模型逐 Token 生成输出，每步只需处理新 Token，已有 Token 的 KV 从缓存读取。相比 Prefill，这里轻松得多。

**♻️ Prefix Reuse**：下一个请求到来时，检查其 Prompt 前缀是否与缓存中的某个前缀完全匹配，匹配部分直接复用 KV 张量，整个 Prefill 阶段直接跳过。


> [!warning] 关键约束：字节级精确匹配
> 前缀匹配是**字节级精确匹配**。哪怕只有一个 Token 的差异，从该位置起的所有 KV 缓存均告失效。这个约束深刻影响了 Agent 的 Prompt 设计策略——稳定的内容必须放前面，动态的内容必须放后面，顺序错了，缓存就废了。



## 二、各大平台的 Prefix Cache 实现

### 2.1 Anthropic：显式控制，最高 90% 折扣

Anthropic 的做法是把控制权交给开发者——你通过 `cache_control` 参数亲手标记哪里是"缓存断点"，系统只在这些地方存储 KV 张量。这种显式设计虽然需要多写几行代码，但精确度极高，90% 的折扣也是目前各大平台中最慷慨的。

**Token 阈值（最小可缓存前缀长度）**

|模型|最小缓存 Token 数|
|---|---|
|Claude Opus 4.6 / Sonnet 4.6 / Haiku 4.5|**4,096 tokens**|
|Claude Sonnet 4.5 / Opus 4 / Sonnet 4|1,024 tokens|
|Claude Haiku 3.5|2,048 tokens|

**定价对比（每百万 Token）**

|模型|标准输入|缓存写入（5分钟）|缓存读取（命中）|
|---|---|---|---|
|Claude Opus 4.6|$5.00|$6.25|**$0.50（节省90%）**|
|Claude Sonnet 4.6|$3.00|$3.75|**$0.30（节省90%）**|
|Claude Haiku 4.5|$1.00|$1.25|**$0.10（节省90%）**|

每个请求最多设置 **4 个缓存断点**，TTL 默认 5 分钟（可升级为 1 小时，费用 2× 写入价）。缓存命中不占用速率限制配额，实际可用吞吐量最高提升 **5 倍**。

### 2.2 OpenAI：全自动，无需任何代码改动

OpenAI 选择了另一条路：什么都不用你做，系统自动搞定。对 1,024 Token 以上的请求，API 自动以 128 Token 为增量单位缓存最长匹配前缀。命中节省约 50% 输入费用，TTL 为 5–10 分钟不活跃后失效（Extended 模式可达 24 小时）。

代价是精细控制能力较弱，省钱幅度也低于 Anthropic。但零代码改动的门槛让它成为最快落地的选择。

### 2.3 vLLM：哈希链式块池（APC）

vLLM 的 Automatic Prefix Caching（APC）采用**哈希链式扁平块池**架构：

- 所有 KV Cache 内存预先分配为固定大小的块（每块 16 个 Token）
    
- 每个块以哈希值标识：`hash(父块哈希 + 本块Token ID序列)`，形成哈希链
    
- 全局哈希表映射哈希值到物理 `KVCacheBlock` 对象，多请求可共享同一物理块
    
- 默认哈希算法：SHA256，防碰撞
    
- 淘汰策略：LRU（仅对引用计数为 0 的块）
    

```Prolog
# 启用 APC
llm = LLM(model="your-model", enable_prefix_caching=True)
```

APC 仅缩短 Prefill 时间，不影响 Decode 速度。新版本已默认启用。

### 2.4 SGLang：RadixAttention（基数树注意力）

SGLang 的 RadixAttention 是目前学术界最精巧的 Prefix Cache 实现之一，发表于 NeurIPS 2024。它用**基数树（Radix Tree）**替代 vLLM 的哈希表，带来更灵活的前缀管理：

- 基数树以压缩 Trie 结构存储，边标签为**变长 Token 序列**（vLLM 固定 16 Token 块）
    
- 匹配粒度为 **Token 级别**，前缀匹配更精细
    
- 天然支持对话树分支：多用户从相同系统提示出发的不同对话路径共享根节点
    
- 调度策略：优先调度具有最长前缀匹配的请求（深度优先），最大化缓存命中率
    
- 完全透明，零配置，无命中时开销可忽略
    

**SGLang vs vLLM 对比**

|维度|vLLM APC|SGLang RadixAttention|
|---|---|---|
|数据结构|哈希表（扁平）|基数树（层次化）|
|匹配粒度|16 Token 块|Token 级别|
|配置要求|需要开启|始终开启|
|多分支对话|较弱|天然支持|

**性能数据**：SGLang RadixAttention 在共享前缀工作负载上实现最高 **6.4× 吞吐量提升**，缓存命中率在基准测试中达到 50%–99%。


## 三、Prefix Cache 对 Long Horizon Agent 的核心价值

### 3.1 什么是 Long Horizon Agent

Long Horizon Agent（长链路智能体）是能够在单次任务中执行 **30–50 步甚至更多工具调用** 的 AI 系统，典型场景包括：

- **深度研究 Agent**：自主搜索、阅读、综合多源信息
    
- **代码 Agent**：理解代码库、执行多轮调试、提交 PR
    
- **工作流自动化**：跨系统协调、长时间任务规划执行
    

与简单 Agent 的根本区别在于：每一步工具调用后，模型需要处理的上下文持续增长——系统提示词 + 所有历史操作记录 + 所有工具返回结果，全部拼在一起作为下一步的输入。

### 3.2 没有 Prefix Cache：成本爆炸 💸

想象一个场景：你雇了一位顾问帮你做市场研究。

每次你问他一个新问题，他都会说："稍等，让我把我们之前所有的对话记录、所有的资料、所有的分析报告，从头到尾完整地重读一遍——然后再回答你。"

你问了 40 次，他重读了 40 遍。你的钱，就这样没的。

这正是没有 Prefix Cache 时，Long Horizon Agent 在做的事情。

---

考虑一个典型场景：

- 系统提示词（含工具定义）：**20,000 tokens**
    
- 任务执行步数：**40 步**
    
- 平均每步新增上下文：500 tokens
    

**不使用 Prefix Cache 时**，每次 API 调用都要重新处理全部前缀：

```Plain
第1步：20,000 tokens 完整 Prefill   ← 全部重读
第2步：20,500 tokens 完整 Prefill   ← 又一次，全部重读
第3步：21,000 tokens 完整 Prefill   ← 还是全部，从头读起
...
第40步：39,500 tokens 完整 Prefill  ← 第四十次，全部重读
总计输入 Token：约 1,190,000 tokens
```

账单来了。以 Claude Sonnet 4.6 定价（$3/M tokens）计算，**单次任务成本约 $3.57**。

听起来还好？现在想想：你有 1,000 个用户，每人每天跑 5 次任务。那是 **$17,850/天**，一个月 **$535,500**。规模一上去，这个数字会让你睡不着觉。

**打开 Prefix Cache（90% 命中率）之后**：

```Plain
缓存命中部分：1,071,000 tokens × $0.30/M = $0.32
未命中部分：  119,000 tokens × $3.00/M  = $0.36
总计约 $0.68，节省约 81%
```

同样的任务，**成本从 $3.57 降到 $0.68**。不是小优化，是量级差异。

研究论文《Don't Break the Cache》（arXiv 2601.06007）实测数据：

在 OpenAI、Anthropic、Google 三大平台上，Prefix Cache 为长链路 Agentic 任务带来**41%–80% 的成本节省**，Time-to-First-Token 改善 **13%–31%**。

### 3.3 延迟：你等的不是回复，是模型在重读课本 📖

每次你的 Agent 完成一步工具调用，准备进行下一步时，模型要做什么？

**没有 Prefix Cache 时**：它要把整本"课本"从第一页翻到最后——系统提示词、所有历史操作、每一条工具返回结果，一字不差地重新过一遍，然后才能开口说话。上下文增长到 30,000 tokens？在标准 GPU 上，这一次"重读"就要花掉你 **10–16 秒**。

40 步任务，每步等 10 秒以上。用户盯着转圈的 loading 动画，不知道发生了什么，只能等。

**打开 Prefix Cache 之后**：模型只需要处理这一步新增的几百个 token——就像一个真正记住了之前所有内容的顾问，直接从上次停下的地方继续。响应时间降至 **1–2 秒**。

生产级数据更能说明问题：llm-d 的分布式调度基准测试显示，启用缓存感知路由（Cache-Aware Routing）后，P90 TTFT 从 **31 秒降至 0.54 秒**，整整提升 **57 倍**。




### 3.4 输入/输出 Token 比值：Prefix Cache 的杠杆效应 🔧

Long Horizon Agent 的输入/输出 Token 比值通常超过 **100:1**。你每花 100 块钱调用 API，其中 90 块钱是在让模型反复读它已经读过的东西。Prefix Cache 就是把这 90 块钱的账单砍掉大半。
![[Pasted image 20260422144612.png]]


### 3.5 Cache 命中率：最关键的生产指标 📊

> "KV Cache 命中率是生产级 AI Agent 最重要的单一指标，直接决定延迟和成本。" —— llm-d 团队

**命中率计算**：`缓存命中 Token 数 / 总输入 Token 数`

|场景|可达命中率|
|---|---|
|固定系统提示词 + 多轮对话|80%–95%|
|RAG（相同文档，多次查询）|70%–90%|
|多 Agent 共享相同系统提示|90%–99%|
|动态内容混入前缀|5%–30%（糟糕）|

### 3.6 Cache 失效的常见陷阱 ⚠️

以下操作会破坏 Prefix Cache，导致命中率从 90% 骤降至个位数：

- 在系统提示词开头插入**时间戳或请求 ID**
    
- 将动态用户信息放在**静态工具定义之前**
    
- JSON 序列化时**Key 顺序不稳定**（Go/Swift 等语言的 Map 默认随机顺序）
    
- 对话历史**非追加式修改**（重排序、中途压缩）
    
- 工具定义**动态变化**（如根据权限动态添加/删除工具）



## 四、主流 Agent 框架中的 Prefix Cache 实践

### 4.1 OpenClaw：从踩坑到成熟的血泪史

OpenClaw 是一个开源个人 AI 助手框架，支持跨平台本地运行（macOS/iOS/Android/Windows），通过多步工具调用执行复杂任务。

**核心策略：稳定前缀 / 动态后缀分离**

OpenClaw 将系统提示词分为两个区域：

- **稳定前缀**：工具定义、技能元数据、工作区文件等静态内容，字节级别保持一致
    
- **动态后缀**：心跳时间戳、每轮元数据等变化内容
    

这个边界被称为 **cache-prefix boundary**，在 Anthropic、OpenAI、Google 等多个 Provider 上统一应用。

**实际效果（GitHub Issues 记录）**

|指标|优化前|优化后|
|---|---|---|
|每次会话 Prompt Token 消耗|~800K tokens|~200K tokens|
|后续轮次响应延迟|10–16 秒|1–2 秒|
|Cache 命中率|84%|~95%+|


**🩸 踩坑实录：从 Issue 里读出的教训**

表格里的数字看起来很美好，但这些数字背后，是团队在 GitHub Issues 里留下的一地鸡毛。

**Issue #49700 ——"我的 Cache 呢？"**有用户发现，明明配置了 Cache，Anthropic 那边的命中率只有可怜的 **~10%**。排查了半天才找到元凶：一个心跳时间戳被混进了系统提示词的"稳定"区域。每隔几秒，时间戳变了，整个前缀就跟着失效了。Cache 当然形同虚设。

**Issue #23715 ——"这个月的账单是怎么回事？"**Cache 策略悄悄失效，没有任何报错，用户浑然不知，每轮对话莫名其妙地慢了，直到月底收到账单——**API 成本高出了整整 5 倍**。钱没了，才意识到 Cache 早已失效。

**Issue #40256 ——"排序这件小事"**系统提示词的几个 Section 顺序写错了。就这么一件"小事"，直接破坏了本地模型的 Prefix Cache 机制。Prefix Cache 是逐字节比对的，把段落顺序换一换，在模型眼里就完全是一份新的提示词，之前缓存的一切全部作废。

**Issue #21999 ——"提示词胖到 Cache 都扛不住"**系统提示词规模膨胀到 **150K Token 以上**，触发了专项性能优化。这个 Issue 的存在本身就说明：当 Agent 能力越堆越多，系统提示词会以你意想不到的速度膨胀，直到撑破天花板。

> 💡 这些踩坑记录最终凝练成了 cache-prefix boundary 这个核心设计——把"永远不变的"和"每次都在变的"彻底隔离开，从源头堵死缓存失效的可能。

### 4.2 Hermes（NousResearch）：用满每一个断点

Hermes 是 NousResearch 开发的开源通用 AI 编程与生产力 Agent，拥有目前最详细的 Cache 架构文档。

**"system_and_3" 策略：用满 Anthropic 的 4 个缓存断点**

Hermes 在 `agent/prompt_caching.py` 中实现了 `apply_anthropic_cache_control()` 函数，对每个请求精确设置 4 个缓存断点：

| 断点位置 | 内容           | 缓存效果            |
| ---- | ------------ | --------------- |
| 断点 1 | 系统提示词        | 全会话共享，命中率 ~100% |
| 断点 2 | 倒数第 3 条非系统消息 | 滚动窗口缓存          |
| 断点 3 | 倒数第 2 条非系统消息 | 滚动窗口缓存          |
| 断点 4 | 最后一条非系统消息    | 为下轮预热           |

**📸 冻结快照模式（Frozen Snapshot Pattern）**

想象一下，你在旅行开始前给全家人拍了一张合照，然后把相机收起来——接下来不管旅途中发生什么，这张合照始终是那个时刻的定格，不会因为路上谁换了件衣服就重拍一遍。

Hermes 的"冻结快照模式"就是这个逻辑：**会话开始时，把系统提示词拍成一张"照片"封存起来**，整个会话期间绝不重拍、绝不修改。

- 临时系统提示词（ephemeral system prompts）需要插入？单独处理，绝不动这张照片
- 中途的记忆注入（memory injection）要更新？另辟蹊径，主前缀纹丝不动
- 上下文压缩触发了？压缩的是对话记录，系统提示词那张"照片"**完整保留**

正因为这张"照片"从始至终保持原样，模型每次看到它都能说一声"我认识你"，Cache 命中率才能稳定在接近 100%。

**效果**：Hermes 通过该策略实现多轮对话 **~75% 输入 Token 成本节省**。



### 4.3 SWE-agent（Princeton NLP）：意想不到的并行天花板

SWE-agent 是专注于自动解决 GitHub Issue 的代码 Agent，是 SWE-bench 系列基准的重要参赛者。

## 六、最佳实践：如何最大化 Cache 命中率

### 6.1 Prompt 结构：稳定在前，动态在后

![[Pasted image 20260422154658.png||300]]


**黄金法则**：静态内容在前，动态内容在后。绝不在静态内容之前插入任何动态信息（时间戳、用户 ID、请求号等）。

### 6.2 工具定义的缓存

工具定义通常是 Agent Prompt 中最大的静态块（生产系统中可达 20,000+ Token）。正确做法：

- 在工具列表的**最后一个工具**上打缓存标记（Anthropic）
    
- 工具 Schema 使用**稳定的 Key 排序**（JSON `sort_keys=True`）
    
- 避免动态添加/删除工具——若需限制工具，使用 `tool_choice` 参数而非修改工具列表
    

### 6.3 多副本部署的路由策略

在多个推理服务实例的生产部署中，需要确保**相同会话的请求路由到相同实例**（Sticky Routing）：

- OpenAI：使用 `prompt_cache_key` 参数
    
- 自建推理服务：基于会话 ID 或前缀哈希做一致性路由
    

实测：通过 `prompt_cache_key` 路由优化，命中率从 60% 提升至 **87%**。

### 6.4 会话内的上下文管理

|实践|原因|
|---|---|
|对话历史只追加，不修改|修改历史破坏哈希链|
|上下文压缩时保留系统提示词完整|确保最大的静态块命中率不受影响|
|工具结果不必全量缓存|动态工具结果频繁变化，缓存反而增加开销|
|使用 Anthropic extended TTL（1小时）用于长任务|避免长耗时任务中途 Cache 过期|

### 6.5 经济测算：Break-even 分析

Anthropic 的缓存写入额外收费 1.25×。**盈亏平衡点**：只需 **1.4 次缓存读取**即可回收写入成本。

对于 Agent 场景——系统提示词会被读取数十上百次——投资回报极为显著。


## 七、结语：不是优化，是生存

Prefix Cache 不是一个可以"之后再考虑"的性能优化，而是 Long Horizon Agent 能够在生产环境中**经济可行、用户可接受**地运转的前提条件。

没有它，Agent 每走一步都在为自己已经走过的路付双倍的钱，还要让用户等它"重读课本"。有了它，那些被浪费的 90% 计算资源重新归位——钱花在刀刃上，速度也回到了它该有的样子。

OpenClaw 的踩坑血泪、Hermes 的冻结快照、SWE-agent 的并行天花板——这些框架走过的弯路，都是值得借鉴的前车之鉴。

> **成本维度**：41%–80% 的整体任务成本节省，90% 的单 Token 价格折扣。
> 
> **延迟维度**：TTFT 从 31 秒降至 0.54 秒，57 倍的速度差距。
> 
> **工程维度**：Prompt 结构、工具排序、路由策略——三个工程决策，直接决定系统能否跑起来。
