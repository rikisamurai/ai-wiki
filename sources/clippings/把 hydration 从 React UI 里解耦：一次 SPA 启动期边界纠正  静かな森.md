React SPA 的冷启常有一种特定视觉故障：业务内容先闪一下，又退回 loading，再过一两秒才稳定。这不是偶发的渲染抖动，它几乎总是同一个根因——持久化数据的 hydration 被绑在了 React UI 的生命周期上。Provider、effect、子树挂载这些 React 内部的调度，被悄悄当成 hydration 的触发器。两条本该分开的时间轴，就在这一刻被强行绑死。

这篇文章只讲一件事：怎么把 hydration 从 React UI 里解耦出去，让 UI 只消费已经准备好的运行时状态。围绕这条主线，再讲三个解耦之后才会浮出来的引申问题——bootstrap 不该 await render、import 副作用、首屏旧指标。它们不是独立的「边界」，而是「这一刀切下去之后，必然掉出来的副产品」。

## 核心：把 hydration 从 React 组件树里搬出来

### 模式

SWR、React Query、Redux Persist 这一类需要从持久化存储「水合」内存状态的库，最常见的接入位置是 Provider。理由非常自然——Provider 是「上下文准备」的天然落点，hydration 看起来就是上下文的一部分。同样自然的还有「在根组件的 useEffect 里发起一次 hydration」「在某个高层布局组件里 await 完再渲染子树」。

这些写法的共同点：把 hydration 的发起时机绑在 React 内部的调度上。Provider 何时挂载、effect 何时运行、子树何时进入 commit，都由 React render 流程决定。而 hydration 的「完成时机」由持久化层（IndexedDB、localStorage、远端预取）的 I/O 决定。两条时间轴被绑在一起，UI 就成了 hydration 的隐式触发器与等待者。

这就是边界放错的位置。

### 失败模式

典型链路：

1. React 渲染 Provider，Provider 启动 hydration（异步打开 IndexedDB）。
2. 子树同步挂载，订阅 SWR key。
3. cache 还没回来，订阅命中空值，触发 fetch，子树跌入 Suspense fallback。
4. hydration 完成，cache 注入，子树再次解析，业务 UI 回归。

### 修法：把 hydration 提升到 app bootstrap 层

解耦的落地形态是：hydration 不再由任何 React 组件触发，而是放到一个明确的 app bootstrap 层，在 createRoot 之前发起；React tree 完全不参与「什么时候开始水合」的决策，它只通过一个原子 gate 决定「业务 UI 什么时候可以显示」。

TS

```ts
export const startAppInitialization = () => {
  if (started) return;
  started = true;

  startSWRCacheScopeSync();
  startImportSettingsFromUrl();
  registerBuiltinToolSurfaces();

  void initializeApp()
    .catch((error) => {
      console.error('[SPA Initialize] failed', error);
    })
    .finally(() => {
      flushSync(() => {
        setAppReady(true);
      });
      startPostRenderInitialization();
    });
};
```

SWR provider 同步提升为 app 级 singleton，hydration 由 bootstrap 调用，SWRConfig 只消费、不再决定时机：

TS

```ts
export const appSWRCacheProvider = swrCacheProvider(getCacheScope);

export const hydrateAppSWRCache = async (): Promise<void> => {
  await appSWRCacheProvider.hydrateScope?.();
};
```

### 可带走的判断

看到任何把 hydration 写进 Provider / useEffect / 子树挂载流程里的代码，先问一句：这份数据的「可用时刻」是不是被这段 UI 代码的「挂载时刻」隐式决定了？如果是，UI 就成了 hydration 的隐式触发器，闪现-回退迟早会出现。让 app bootstrap 层托管 hydration，UI 只消费结果。

## 解耦之后掉出来的三个引申问题

把 hydration 从 React UI 里搬出来之后，三件事会立刻浮上来。它们不是独立的「启动期问题」，而是上一节那条解耦切下去之后必然要处理的副产品——一个关于 bootstrap 与 render 的关系，一个关于 import 副作用，一个关于首屏指标。

### 引申一：bootstrap 不该 await React render

建立了 bootstrap 层之后，第一个错位就出现在它和 React render 的衔接上。最常见的写法是：

TS

```ts
await initializeApp();
createRoot(root).render(<App />);
```

这个形状把「需要等待」和「必须串行」混为一谈。异步 I/O 期间（IndexedDB 打开 + 读取常见 50-200ms），主线程完全空闲——本可以在这段时间里并行完成 React 模块求值、router 构建、provider chain 解析，结果都排到了 I/O 之后。

修法：两条线并行启动，用一个原子 gate 协调可见性：

TSX

```tsx
startAppInitialization(); // 同步返回，内部 void 异步任务

const router = createAppRouter(webRoutes);

createRoot(document.getElementById('root')!).render(
  <ThemeProvider>
    <RouterProvider router={router} />
  </ThemeProvider>,
);
```

关键在「不显示」≠「不解析」。React tree 在 hydration 没完成时就开始解析，路由、布局、provider 全部就位。AppLayer 按 appReady 原子决定渲染 loading screen 或业务子树：

- I/O 线：IndexedDB 打开、cache 读出、塞回 in-memory map。
- CPU 线：React 模块求值、router 构建、provider chain 解析。
- Hydration 完成的微任务里 flushSync(() => setAppReady(true))，业务 UI 直接出现。

flushSync 在这里是必要的：它保证 ready 状态在同一个微任务内同步切换，避免「ready 已设但下一帧才渲染」造成的可见空白。

判断：「等待」与「串行」是两件事；可见性也不等于解析。任何 bootstrap 阶段都要问：这段 await 期间，React tree 是否本可以同时被解析？能并行的工作，不要串行。

### 引申二：解耦把 import 顺序改了，全局副作用浮出水面

解耦动作里有一步是把原本散在 React tree 内的全局注册搬到 bootstrap 顶部统一调起。这一搬，模块图的求值顺序就变了。如果项目里存在「import 一个 sub-package 时顶层就把自己注册到全局 map」的写法——也就是注册依赖 import 行为而不是依赖显式调用——bootstrap 顺序立刻把原本被掩盖的环暴露出来。

这类写法在很多 SPA 里都存在。tool registry、route registry、handler registry 的常见形态就是：

TS

```ts
// 子包顶层
import { surfaceRegistry } from '@/registry';
surfaceRegistry.register('my-tool', { /* ... */ });

export const myExecutor = () => { /* ... */ };
```

调用方只需要 import './my-tool'，注册自动发生。看起来很优雅，实际把模块图的求值顺序变成了运行时正确性的一部分。LobeHub 这次解耦动作刚跑起来，控制台立刻给出了：

TEXT

```
Uncaught ReferenceError: Cannot access 'AgentManagerRuntime' before initialization
    at executor.ts:40:17

Uncaught ReferenceError: Cannot access 'agentBuilderExecutor' before initialization
    at index.ts:135:3

Uncaught ReferenceError: Cannot access 'credsExecutor' before initialization
    at index.ts:140:3
```

这些错误的根因不是新代码引入的 bug。这是一个一直存在但被 import 顺序掩盖的设计错误：模块 A 顶层引用了模块 B 顶层的 const，B 又通过链回到 A，B 求值时 A 还没把那个 const 写进 module record。原本可工作，仅仅是因为 React tree 内部的 import 顺序碰巧让 A 先于 B 求值完成。解耦换了 entry，环就浮出来。

第一反应通常是把出问题的 import 改成动态加载 const mod = await import('./my-tool')。这是伪修法——它只把炸点推后，副作用仍然挂在模块顶层、所有权仍然挂在「import 时」。一旦另一条调用路径在更早的时间点拉进同一个包，TDZ 又回来了。触发时间不等于所有权。

修法：把「import 时做的事」改成「export 一个 register() 函数，由 app 在 bootstrap 显式调用」：

TS

```ts
// 子包：副作用退到函数体内
import { surfaceRegistry } from '@/registry';

export const registerMyTool = () => {
  surfaceRegistry.register('my-tool', { /* ... */ });
};

export const myExecutor = () => { /* ... */ };
```

TS

```ts
// app bootstrap：一处统一调度
import { registerMyTool } from '@lobechat/my-tool/register';

export const registerBuiltinToolSurfaces = () => {
  registerMyTool();
  // ... 其他显式注册
};
```

这样改之后，「谁负责注册」和「何时注册」都是一阶问题，不再由 bundler 的求值顺序间接决定。这是 LobeHub 这次顺手做的全包扫描：

| 子包 | 旧形态 | 新形态 |
| --- | --- | --- |
| builtin-tools/src/{inspectors,renders,portals,placeholders,interventions,streamings}.ts | 顶层调用各子包，import 时写入 surface 全局 map | 新建 register.ts，registerBuiltinToolSurfaces() 显式注册 |
| builtin-tool-{knowledge-base,lobe-agent,local-system,task}/src/client/index.ts | client barrel 同时拉出 UI 和 executor，UI 路径带出 executor 副作用 | client 只导出 UI；executor 独立 export |
| store/tool/slices/builtin/executors/index.ts | import 时把 executor 注入 map | 导出 registerBuiltinToolExecutors()；invokeExecutor() 做兜底 |

Whiteboard

判断：看到一个包顶层有非纯计算（数组 push、map set、订阅注册、副作用函数调用），就要警觉。一阶问题是：这件事谁拥有？什么时候应该发生？把答案落在显式函数里，而不是 import 行为里。这一条规则对应的不只是 SPA 启动期；它对任何会被 bundler 重排序的代码都适用。

### 引申三：旧首屏指标无法证明解耦真的有效

解耦了之后，需要测「业务首屏」是否真的更稳定。常见的指标是「#loading-screen 节点是否还在」「\[data-app-loading\] 是否消失」。它符合直觉——那是肉眼看到的第一层过渡。但这种指标无法捕捉 Suspense 回退。loading-screen 卸下之后，Suspense 完全可以接着进来制造下一次「loading 状态」。当文章上半节的 hydration 还在错位时，闪现-回退就发生在这个观察盲区里——指标变绿，体感更差。

解耦之后，指标也必须跟着升级。否则你证明不了改造确实修掉了「闪现-回退」，只能证明 loading-screen 消失得更早一点。

修法：定义「稳定业务首屏」时必须包含一个无回退的稳定窗口：

- #root 可见
- 导航面板出现业务文本
- 首页主输入区出现
- #root 可见文本达到业务阈值
- #loading-screen 不存在
- 满足以上条件后连续 1500ms 不回退

最后一条是关键，它把 Suspense 回退捕获在指标定义里——任何「闪现一下」的退化都会被算成未达到。

判断：首屏指标必须能区分「出现」与「稳定」。如果你的指标在 fallback 回退过程中给出绿灯，它在度量的是加载动画，而不是用户感受到的应用就绪。

## 基线方法：测得准比测得多重要

解耦的核心收益是「闪现-回退」消失，这件事必须被基线数据证实，不是凭体感。但启动期的基线测量本身有两个隐性陷阱，踩中任何一个都会让数字失去意义。

### 陷阱一：dev server 不算数

Vite dev server、HMR、source map、未压缩的依赖、未 tree shake 的图，都会污染指标。必须在生产构建产物里测：

BASH

```bash
npm run package:local --prefix=./apps/desktop
```

### 陷阱二：worktree 的 node\_modules 会跨版本污染

如果直接在当前 worktree 上 git checkout 旧分支跑「旧」基线，node\_modules 和 workspace symlink 仍然指向当前的包源代码。「旧分支」的应用代码在运行时通过 symlink 拉到的是 改造后的 共享包。基线测的是两个版本的 app 代码 + 同一份新的包代码，完全无效。

用 git worktree 开干净环境，独立装依赖：

BASH

```bash
git worktree add --detach /tmp/baseline-old origin/canary
cd /tmp/baseline-old
pnpm install

# apps/desktop 不在根 pnpm-workspace packages 里，单独装
cd apps/desktop
pnpm install
```

仓库根.npmrc 若配了 lockfile=false，--frozen-lockfile 在这条路径上不是正确选项。
