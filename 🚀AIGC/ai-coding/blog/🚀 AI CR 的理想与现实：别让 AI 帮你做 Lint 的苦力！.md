# 🚀 AI CR 的理想与现实：别让 AI 帮你做 Lint 的苦力！

你的团队刚刚接入了 AI Code Review。第一个 PR/MR 提交后，AI 留下了 15 条评论。

”太强了，一下子发现这么多问题！”

你点开评论，逐条看下去：

- “建议使用 `const` 替代 `let`”
- “此处 `console.log` 不应保留在生产代码中”
- “条件语句中调用 `useState` 违反了 Hooks 规则”

看起来很专业，对吧？

但等一下——这三条问题，ESLint 在你==保存文件的那一秒==就能全部拦截。不需要等到 PR 阶段，不需要消耗 token，不需要 AI。

AI Code Review 火了。从 GitHub Copilot 的 Code Review 到各种开源方案，AI CR 正在成为团队工程效能建设的”标配”。似乎不接入一个 AI CR 工具，就意味着你的团队落后了。

但冷静下来想想——它真的在解决问题吗？还是“新瓶装旧酒”，我们在==用新工具重复解决旧问题==

## ❌ AI CR 的局限性

![[Pasted image 20260316213714.png]]

### 1. 发现时机晚

AI CR 最根本的问题是：==它在错误的时间做检查==。

来看两个真实案例：

![[Pasted image 20260316204218.png]]

AI CR ：这行 `console.log` 没有与错误监控系统配合使用，生产环境中不应该保留调试日志……但 ESLint 的 `no-console` 规则，一行配置就能在你==保存文件的那一秒==拦截这个问题。



![[Pasted image 20260316202725.png]]

再看这条——AI CR 将其标记为 **CRITICAL** 级别，指出条件语句中调用 `useState` 违反了 React Hooks 规则。但实际上 `eslint-plugin-react-hooks` 的 `rules-of-hooks` 规则在编码阶段就会直接报错。

这两个问题的共同点：它们本不该暴露到 CR 阶段。Lint 在保存文件时就能拦截，`tsc` 在编译时就会报错。这些问题跑到了 CR 阶段才被 AI"发现"，说明的不是 AI 厉害，而是你的==工程基建有缺失==。

用 AI CR 的"召回率"来证明它的价值，有点像用创可贴的使用量来证明创可贴有用——也许更值得关注的是：为什么会有这么多伤口？

> [!important] Shift Left 原则
> 工程效能的核心理念之一是 **Shift Left**——让问题在尽可能早的阶段被发现和解决。AI CR 恰恰违背了这一原则：它把本该在编码阶段解决的问题，推迟到了 Review 阶段。

### 2. 执行时间长

Lint 的反馈是秒级的——保存文件，编辑器立刻标红。而 AI CR 的流程是：读取 diff → 拉取上下文 → 模型推理 → 生成评论，一个中等规模的 PR 可能需要几分钟甚至十几分钟才能跑完。

这不只是"慢一点"的问题。反馈延迟直接影响修复成本——编码时发现一个问题，改一行就好；等到 PR 阶段才发现，开发者需要重新切回上下文、理解评论、修改代码、再次提交。==反馈越晚，修复成本越高。==

而且当 AI CR 变慢时，开发者会开始"先合再说"——反馈还没出来，PR 已经合了。

### 3. 结果不确定

Lint 规则是确定性的——一段代码要么违反规则，要么不违反，没有"可能违反"。而 AI CR 是概率性的，同一段代码跑两次，结果可能不同。

这种不确定性体现在多个层面：

- **幻觉**：AI 可能标记根本不存在的问题，也可能对真正的风险视而不见
- **Context 限制**：AI 往往只看到一个 diff，缺乏完整的项目结构和业务上下文，导致误判
- **噪音**：大量低价值评论（"变量名可以更具描述性"、"建议加个空行"）淹没真正重要的反馈

> [!warning] Review 疲劳
> 当一个 PR 上有 20 条 AI 评论，其中 18 条是"建议改个变量名"，开发者会开始无视所有评论——包括那 2 条真正重要的。

对于代码质量守卫这个角色来说，==确定性是一项基本要求==。一个时灵时不灵的守卫，不如没有。

## ⚖️ 核心原则：Lint 能做的事，不要交给 AI

把上面三个局限性放在一起看：

| 维度      | 🤖 Lint              | 🧠 AI CR      |
| ------- | -------------------- | ------------- |
| **时机**  | 编码时 / pre-commit(极早) | PR 提交后(偏晚)    |
| **速度**  | 秒级反馈                 | 分钟级           |
| **确定性** | 确定性结果，通过或不通过         | 概率性输出，伴随幻觉和噪音 |
| **成本**  | 几乎为零                 | 按 token 计费    |

每一个维度，Lint 都完胜。Lint 这些工具在这个领域深耕了十年以上，规则明确、执行确定、几乎零成本。能用 Lint 解决的问题，就不要让 AI CR 来做。

## ✅ 正确的姿势：让 AI 各司其职

不是说 AI 在代码质量领域没有用，而是要让它在对的位置发挥价值。

### AI 写 Lint：确定性问题交给确定性工具

团队有特殊的编码规范？比如"所有 API 请求必须经过统一的 request 封装"、"禁止在组件中直接调用 localStorage"——这些规范以前很难用 Lint 表达，只能靠 CR 时人肉检查，现在可以让 AI 帮你生成 ESLint 自定义规则。

流程很简单：把规范描述给 AI → AI 生成 ESLint 规则 → 加入项目配置 → 接入 pre-commit hook 和 CI。**一次编写，永久生效。**

已经有工具在做这件事了。比如 [eslint-gpt](https://github.com/ycjcl868/eslint-gpt)——你只需要输入示例代码，它就能自动生成对应的 ESLint 规则，支持保存和分享。核心理念很简单：==让 AI 生成规则，而不是让 AI 反复检查==。AI 的算力花一次，Lint 规则跑一辈子。

对比一下两种方案：
- **AI CR 方案**：每次提 PR，AI 都"提醒"一遍"这里没有用统一封装"——开发者可能忽略，AI 也可能漏掉
- **AI + Lint 方案**：规则写好后，编码时编辑器直接报错，提交时 hook 直接拦截——确定性 100%，零遗漏

同样是用 AI，一个是让它==反复做苦力==，一个是让它==一次性解决问题==。

### AI CR：聚焦私域业务逻辑

AI CR 不是没有用武之地，但应该用在真正需要"理解"的场景——那些 Lint 规则根本无法表达的问题：

- **结合 PRD + Test Case 分析业务风险点**：比如本次需求涉及支付流程改动，AI 结合 PRD 和测试用例，分析是否有遗漏的边界条件——这是 Lint 完全做不到的
- **审查难以规范化的规则**：比如"并发场景下是否有竞态风险"、"这个缓存策略在高流量下是否合理"——这类问题无法写成 Lint 规则，需要上下文理解

AI CR 的真正价值在于==私域业务逻辑==，而不是通用代码规范。把 AI 的算力花在"检查 console.log"上，是一种浪费。

> [!note] 理想与现实
> 目前 AI CR 在业务逻辑审查上的表现还在发展中——它拿到的往往只是一个 diff，缺乏完整的业务上下文。但方向是对的，值得探索。关键是==别让它做 Lint 的活==。

### 成熟的工程手段兜底

不要忘了那些已经存在十年以上的成熟方案：

- **pre-commit hook**：提交前自动运行 Lint，不通过就不让提交
- **CI Lint 检查**：PR 级别的 Lint 卡点，确保没有规则违反合入主干
- **Editor 插件**：编码时实时提示，问题发现在最早的时刻

它们更快、更准、更便宜，是代码质量的基本盘。

## 回归本质：让问题在最早的阶段被解决

工程效能的核心理念很朴素：让问题在==最早的阶段==被发现和解决。

如果我们把本该在编码阶段解决的问题推迟到了 Review 阶段，然后用 AI 的"召回率"来庆祝，那方向可能走偏了。

AI 是好工具，但好工具也要用对地方。与其让 AI 在 CR 阶段做事后检查，不如让它帮你写出更好的 Lint 规则，把问题扼杀在萌芽阶段。与其追求 AI CR 的召回率，不如先把 Lint 基建体系搭好——这往往是投入产出比更高的选择。

扁鹊说长兄医术最高——长兄治病于未发之前，世人不知其能；扁鹊自己治病于重症之时，反而名满天下。AI CR 就像扁鹊，在 CR 阶段"发现"问题；而 Lint + 工程基建更像扁鹊长兄，在编码阶段就把问题拦截了。

AI CR 在 CR 阶段告诉你 这里有问题，而 Lint 不会让问题暴露在 CR 阶段
