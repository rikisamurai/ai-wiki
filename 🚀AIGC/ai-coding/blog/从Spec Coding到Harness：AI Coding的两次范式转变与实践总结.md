AI 编程正从”聊天试错”的 Vibe Coding，进化到以”自动化约束”为核心的 Harness Engineering 范式。
AI 编程在过去一年经历了从 Vibe Coding（氛围编程）到 Spec Coding（规约编程），再到 Harness Engineering（约束工程）的两次关键范式转变。Vibe Coding 依赖直觉与 AI 对话，产出成果常止于“大型 Demo”；Spec Coding 试图通过详尽的规约（Spec）指导 AI，却面临规约本身维护困难、易于腐烂的困境，并分化出“规即源码”、“以规为锚”、“先规后码”三种形态。

为解决上述问题，Harness Engineering 应运而生。它不再强依赖于易过时的自然语言规约，而是将重心转向构建一个由**测试用例****、自动化链路、静态检查（****Lint****）和** **CI/CD** **组成的“约束”环境**。在这个框架下，工程师的核心工作从编写业务代码，转变为搭建和维护这个“软件工厂”，让 AI Agent 在明确、可验证的约束下自主完成代码生成、测试、修复和迭代，最终实现更高程度的自动化和更可靠的交付质量。这不仅是对开发流程的重塑，更预示着一种向“算法工程师”模式的转变——从确定性的代码逻辑推理，到非确定性的“调参”，即不断优化约束环境以提升 Agent 的表现。

***

## 两次范式转变：从 Vibe 到 Harness

AI 编程的演进路径揭示了一个核心矛盾：**如何在释放 AI 强大生产力的同时，确保工程的严谨性与最终产物的可靠性。** 

### 第一阶段：从 Vibe Coding 到 Spec Coding

- **Vibe Coding****（****氛围编程****）**：由 Karpathy 提出，核心是“和 AI 边聊边写代码，大概试一试没问题，就交付”。这种方式适合从 0 到 1 的探索性项目或个人工具，但其产出物往往是功能看似完备但细节粗糙的“大号 demo”，难以胜任严肃的商业软件开发。其本质是“许愿编程”，缺乏工程严谨性。
	
- **Spec Coding（规约编程）**：作为对 Vibe Coding 的反思，Spec Coding 强调“先规后码”，试图通过编写详尽的规约（Specification）来指导和约束 AI 的行为。然而，这一流派内部存在巨大分歧，并演化出三种截然不同的实践形态：
	

| 形态                            | 核心理念                                     | 优点                                              | 核心挑战                                                                    |
| :---------------------------- | :--------------------------------------- | :---------------------------------------------- | :---------------------------------------------------------------------- |
| **规即源码** **(Spec-as-source)** | 将 Spec 作为唯一“源文件”，人类只修改 Spec，代码完全由 AI 生成。 | 理想化，追求“一次编写，处处运行”的终极自动化。                        | **不可控性**：自然语言的模糊性导致微小修改就可能产生巨大代码差异；模型迭代也让结果难以复现，是测试验证的灾难。               |
| **以规为锚 (Spec-anchored)**      | 同时维护 Spec 和代码，将 Spec 作为后续迭代的参考。          | 在理想与现实间取得平衡，多数框架（如 Spec-Kit）采用此模式，并提供标准流程与目录规范。 | **伪需求与僵化**：流程可能过于死板，且“切换技术栈”等预设场景未必真实。Spec 的历史记录价值有限，最终仍会过时。            |
| **先规后码 (Spec-first)**         | 仅在单次迭代前编写 Spec，用完即抛，不维护。                 | **务实、易于落地**，适合个人实践，降低了团队协作的门槛。                  | **Spec 的宿命是腐烂**：一次性的 Spec 无法解决长期维护问题，代码依然是唯一的事实标准，过时的 Spec 反而可能误导新人与模型。 |

**Spec Coding 的核心困境**在于 **Spec 的维护成本与生命周期问题**。代码是演进的，而 Spec 作为其快照，几乎注定会过时、腐烂，最终成为“技术负债”。

### 第二阶段：Harness Engineering（约束工程）

面对 Spec Coding 的三大难题（人工测试/Review 耗时、Spec 遵守性、Spec 维护），OpenAI 提出了 Harness Engineering，完成了第二次范式转变。其核心思想是：**与其依赖易变的自然语言 Spec，不如构建一个刚性的、自动化的“约束”环境。** 

Harness（约束/治理）的本质，是构建一个能约束 Agent 行为、引导其朝正确方向迭代的环境。工程师的角色从“代码工人”转变为“软件工厂的设计者与维护者”。

“约束工程”认为，”测试用例是最好的文档”。一个由 CI/CD 保证的测试用例集，远比任何静态的 Markdown 文档都更可靠、更与时俱进。

> [!important] 约束悖论（The Constraint Paradox）
> Harness Engineering 有一个反直觉的核心结论：**赋予 Agent 更高的自主性，需要的是更严格的约束，而不是更宽松的约束。**
>
> 人们往往认为给 AI 更多自由能激发更大潜力。但实践表明，缺乏约束的 Agent 不是在”自由发挥”，而是在随机漫步。边界越清晰，Agent 在边界内的执行越可靠。这正是 Harness 的本质：用刚性约束换取真正的自主。

**Harness 不是预先设计的，而是被动反应式生长的。**

正确的工作方式是：**每当 Agent 犯了某类错误，就把”让这个错误永远不再发生”这件事工程化**——写成测试、Lint 规则或 CI 检查。Harness 是通过积累失败经验逐渐生长出来的，而非在项目开始时一次性预设完整。

用一个比喻：Harness 不是盖房前画好的图纸，而是每次漏水后加装的防水层，每次火灾后加装的烟雾报警器——是对现实摩擦力的工程化回应。

## 方法论与工程实践：构建你的“约束工厂”

Harness Engineering 并非一个抽象概念，而是由一系列具体的工程实践构成。它旨在打通“AI 开发-反馈-修复”的自动化闭环。
**核心实践**

- **完备的****测试用例****是基石**：在开发初期就与 AI 共同构造覆盖核心场景的单元测试和集成测试。这些测试用例不仅是验证标准，更是 AI 理解需求的“活文档”。
	
- **打通****自动化测试****链路**：让 AI Agent 具备自行运行测试、读取测试报告、分析日志和 Metrics 的能力。这是实现自我迭代的前提。
	
- **强制的架构与规范约束**：利用定制化的 Lint 工具强制执行架构分层、代码规范等“硬规则”。当 Agent 违反规则时，工具能提供明确的反馈，引导其自行修复，这远比在 Prompt 中反复强调有效。
	
- **CI/CD** **不再是“周边”**：CI/CD、单元测试等过去被视为“辅助”的设施，在 Harness 模式下成为项目的核心基础设施，需要从一开始就投入精力建设。
	
- **Agent 的自主迭代与闭环**：AI 的任务不止是生成代码。它需要在一个循环中不断试错：**编码 → 运行测试/****Lint** **→ 分析失败 → 修复代码 → 再次验证**，直至所有约束都被满足。
	

**前端工程启示**

- **质量保障左移**：将 E2E 测试、UI 组件测试、视觉回归测试等自动化用例作为核心交付物。AI Agent 的产出必须通过这些用例的检验。例如，利用 Playwright 或 Cypress 编写测试脚本，让 Agent 在开发过程中持续运行。
	
- **工具链即约束**：强化 ESLint、Stylelint、Prettier 等工具的规则集，并将其整合到 CI 流程中。为前端项目定制专门的 Lint 规则，如检查是否使用了废弃的组件、API 是否符合规范、模块间依赖关系是否正确等。
	
- **平台化与”黑盒”开发**：对于重复性高的前端开发任务（如活动页、中后台页面），可以构建”软件工厂”平台。工程师定义好页面的 DSL（领域特定语言）或测试用例，AI Agent 在平台内部完成开发、部署、测试，交付的是一个通过验证的线上 URL，而非源码。


**一个常见的验证盲区：行为正确性**

当前大多数 Harness 体系擅长捕捉**结构性错误**（代码能否运行、测试能否通过），但存在一个系统性盲区：**行为正确性**（Behavioral Correctness）——即代码是否真的做了用户期望的事。

以下三类验证在大多数项目的 Harness 中缺失：
- **用户价值 E2E 验证**：核心用户路径是否符合真实使用场景？
- **意图失败检测（Intent Failure）**：代码通过了所有测试，但实现的是错误的需求？
- **对抗性数据集**：边界输入能否暴露隐藏的逻辑错误？

> [!warning] 这不是 Harness 的替代方案，而是 Harness 需要覆盖的下一层。如果你的 Harness 目前只有单元测试和 Lint，行为正确性验证就是你的下一个建设目标。

## 工具与技巧：AI Coding 工具箱解析

原文作者分享了一系列实用工具和技巧，强调了工具组合与特定场景下的最优选择。

- **Ralph Loop：应对长程任务的“重启大法”**
	- **场景**：执行跨时数天甚至数周的超长程任务（如重构）。
		
	- **方法**：通过一个外部循环反复重启 AI Agent（如 Claude Code），将每次的进度和状态持久化到 `progress.txt` 和 `git commit` 中。这避免了因上下文窗口无限增长导致的性能下降和成本爆炸，同时保证了任务的可中断和可恢复性。
		
- **OpenClaw****：云端开发的“****Vibe Coding****”利器**
	- **场景**：快速开发和部署团队内部的提效项目、前端应用原型。
		
	- **优势**：在远程开发机上直接进行“开发-部署-预览”的闭环，所见即所得。用户无需关心部署细节，真正实现“聊天编程，只看结果”。
		
- **Superpowers (Skill-based)：“头脑风暴”与灵活组合**
	- **亮点**：基于 Skill 机制，比固化的 Spec 框架更灵活。其 `brainstorming` 技能通过**反问澄清需求、提供备选方案**，不仅能辅助设计，更能拓宽开发者的技术视野。
		
- **Bytedcli：打通内部研发平台的“瑞士军刀”**
	- **价值**：让 Agent 具备操作内部研发平台（如自动部署、触发测试）的能力，是实现 Harness 自动化链路的关键一环。
		
- **自定义 Skills：弥补模型知识盲区的“外挂”**
	- **问题**：AI Agent 对内部库（如 ByteMetrics, Overpass RPC）缺乏认知，容易产生幻觉。
		
	- **解法**：为这些内部库创建专有的 Skill。通过 Aime 等工具读取内部文档，即可快速生成 Skill，作为模型的“知识外挂”。
		
- **AGENTS.md：项目级上下文的”备忘录”**
	- **原则**：只放无法从代码中直接推断的**关键信息**。
	- **数据支撑（ETH Zurich 研究）**：实验测试了 138 个 agentfile，发现 LLM 自动生成的 agentfile 反而**损害** Agent 表现，代价是额外 +20% 的 token 消耗；人工编写的 agentfile 平均提升也只有约 4%。结论：AGENTS.md 的价值不在于内容多寡，而在于**信息密度**——不要让 AI 帮你写，也不要把它写成百科全书。
	- **内容建议**：
		- **项目部署架构**：各环境、各集群的角色与交互方式。
			
		- **核心概念与关系图**：用最简洁的图或文字定义项目中的核心实体及其关系，确保人与 AI 对话时“频道一致”。
			
- **模型选择：Harness 决定模型表现，而非相反**
	- **反直觉结论**：模型的实际表现高度依赖 Harness 的设计质量。同一个模型，在优化过的 Harness 下排名可以从全局第 33 位跃升至第 5 位。这意味着，在投入精力升级模型之前，先优化 Harness 往往回报更高。
	- **共识**：不存在万能的”统一大模型”，不同模型有不同的擅长领域，应根据任务性质（创意、逻辑、代码、长文）选择最适合的。

## 现实落地：在“屎山”中循序渐进

直接在陈旧项目（“屎山”）中全盘实施 Harness 是不现实的。正确的路径是**循序渐进，从自动化你已有的手动验证行为开始**。

![[Pasted image 20260323191316.png|500]]


**核心思路**：研究你每次开发后的手动测试步骤，然后尝试用 AI 将其自动化。

- **实践案例：从手动验证到自动化回归**
	- **现状**：项目缺乏单元测试，但你每次上线前都会手动在 PPE 环境部署一个泳道，调用几个关键接口，然后去 Argos 查看日志确认无误。
		
	- **自动化第一步**：
		- **用例化**：将你手动测试的步骤，整理成结构化的测试用例（如 toml 格式），包含请求、断言描述等。
			
		- **自动化脚本**：使用 Bytedcli 编写一个脚本，让 AI 在完成代码后，能自动完成“部署到泳道 → 遍历执行测试用例 → 请求接口 → 抓取 Argos 日志”这一系列操作。
			
		- **建立反馈**：让 AI Agent 能够读取测试结果和日志，如果发现异常，则进入下一轮的 bugfix 循环，直到所有用例通过。
			
- **对前端开发的具体建议**
	- **优先自动化 E2E 和视觉回归**：前端项目的“接口”就是用户界面。将核心用户流程编写成 Playwright/Cypress 脚本，作为 AI 开发的“约束”。对于 UI 组件库，建立视觉回归测试，确保 AI 的修改不会破坏现有样式。
		
	- **利用现有工具链**：将前端的 Lint、类型检查、单元测试（Vitest/Jest）等流程接入 AI 的开发循环，使其成为强制约束。
		
	- **逐步构建“页面工厂”**：从最规范、重复度最高的页面类型（如营销活动页）开始试点。为这类页面定义标准的测试用例和交付规范，让 AI 承包端到端的开发，逐步建立信任和沉淀能力。

## 时间管理与衡量指标：重新定义“效率”

- **三维任务划分**：为了最大化利用 AI，作者提出了一个任务分类框架，以便并行处理不同类型的任务。
	

| 任务类型 | 特点 | 适用工具/模式 | 程序员角色 |
| :-- | :-- | :-- | :-- |
| **长程强约束** | 任务复杂，但用例/规范完善（如重构 GCC）。“Agent Native” 型项目。 | Ralph Loop，让 Agent 长期自主运行。 | 监控者，偶尔介入解决死循环等疑难杂症。 |
| **长程弱约束** | 任务不重要，但用例/规范缺失（如内部提效工具）。 | OpenClaw（Web）、Trae Solo（桌面），放手让 AI 发挥，跑偏了重来。 | 产品经理，定义需求，验收结果。 |
| **短程弱约束** | 日常琐碎业务需求，项目重要但规范缺失。典型的“屎山”维护。 | IDE 模式的 Trae，以人类为主导，AI 辅助生成局部代码。 | 主力开发者，对 AI 产出进行详细 Review。 |

程序员可以同时处理这三类任务，例如，在等待“长程强约束”任务（由 AI 主导）运行结果的同时，手动处理“短程弱约束”的日常需求。

- **终极衡量指标：Agent 等待时间**
	- **理念转变**：真正衡量 AI Coding 效率的，不是 Token 消耗量或 AI 代码生成占比，而是 **Agent 因为等待人类反馈而浪费了多少时间**（`human wait`）。
		
	- **目标**：**将 Agent 等待时间降为 0**。这意味着你需要构建一个高度自动化的”约束工程”，让 Agent 能够 24 小时不间断地在”开发-测试-修复”的闭环中自主工作，最大化其运行效率。

## 总结：Harness 的三层成熟度框架

Martin Fowler 将 Harness Engineering 的完整体系归纳为三层，可用于评估你当前 Harness 的成熟度：

| 层次 | 内容 | 对应文中实践 |
| :-- | :-- | :-- |
| **Context Engineering** | 给 Agent 正确的上下文：AGENTS.md、Skills、文档体系 | 自定义 Skills、AGENTS.md 规范 |
| **Architectural Constraints** | 刚性的架构约束：分层规则、Lint、CI/CD | 强制 Lint、CI/CD 作为核心基础设施 |
| **Entropy Management** | 对抗熵增：持续清理 AI slop、行为正确性验证 | 屎山渐进改造、行为验证盲区 |

大多数团队目前处于第一、二层的初级阶段。**第三层 Entropy Management 才是 Harness 能否持续运转的关键**，也是最容易被忽视的一层。

> [!tip] 判断你是否进入了 Harness Mode：
> 当 Agent 犯了一个错误，你的第一反应是”修复这个错误”，还是”工程化一个约束让这个错误永不再发生”？
>
> 如果是前者，你还在 Debug Mode。如果是后者，你已经在 Harness Mode 了。
