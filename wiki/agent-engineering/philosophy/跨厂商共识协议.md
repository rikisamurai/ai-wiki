AI Coding 生态正在涌现一类"跨厂商共识协议"：单一规范被 N 个互相竞争的 vendor 同时实现，从而把"互操作"做成了基础设施而不是私有特性。MCP（工具调用）、Agent Skills（能力分发）、Git AI standard（代码 attribution）是同一类东西——它们的价值不来自单个厂商的实现质量，而来自"足够多人都同意按这个 schema 玩"。

> [!note] 不是技术问题，是协调问题
> 这类协议的难点几乎都不在 spec 本身——MCP 就是一个 JSON-RPC 子集，Git AI standard 就是一份 git notes 的字段约定。难点在于让多个互相竞争的 vendor 同时实现：每多一个 vendor 加入，对所有人价值都增加（网络效应），但谁先实现谁吃亏（先跑的代价）。

## 三个典型案例

> [!example] 同形态、不同领域
> | 协议 | 领域 | 多厂商共识体现 |
> |---|---|---|
> | [[wiki/claude-code/mcp\|MCP]]（Model Context Protocol）| Agent ↔ 外部工具 | Anthropic / OpenAI / Google / Cursor / Cody 等都接 |
> | [[wiki/skills/agent-skills\|Agent Skills]] 规范 | Agent 能力的分发 / 复用 | Anthropic / Cursor / OpenCode 等可加载同一个 SKILL.md |
> | Git AI standard（v3.0.0）| AI 生成代码的 attribution | [[wiki/claude-code/git-ai\|Git AI]] 已对接 12 个 coding agent，spec 公开 |

## 共同特征

> [!compare] 跨厂商协议 vs 私有 SDK
> | 维度 | 跨厂商协议 | 私有 SDK |
> |---|---|---|
> | 价值来源 | 多家同时实现的网络效应 | 单家实现的功能完备度 |
> | 演化模式 | 公开 spec + 版本号（如 v3.0.0） | 厂商单边定义、随产品迭代 |
> | 用户成本 | 学一份 spec 走遍生态 | 每家学一遍 |
> | 锁定风险 | 低 | 高 |
> | 早期采用激励 | 弱（先到 = 不兼容旧 vendor） | 强（独家功能） |

## 为什么这三个会并行出现

> [!important] 时间点不是巧合
> 多 agent 协作 + 多模型混用成为常态后，每个团队/产品都在用 N 个互相不兼容的 agent。这时候三个层级都需要标准：
>
> - **能力分发层**（Skills）：让一份 SKILL.md 在多个 agent 上运行
> - **运行时工具层**（MCP）：让一个工具被所有 agent 调用
> - **事后归因层**（Git AI）：让一份 attribution 跨多个 agent 写出的代码统一查询
>
> 三个协议解决的都是"vendor 多样化产生的互操作摩擦"。

## 实操判断：什么情况下值得自己起一个

如果你正在做新工具，要不要给它配套一份"开放 spec"？

> [!tip] 三个判别条件
> 1. **是否会被多 vendor 用**：单产品独占场景不需要 spec
> 2. **schema 是否稳定**：还在快速演化的字段不要早早冻结
> 3. **能否拉到 ≥ 2 个不同阵营的实现者**：少于 2 个 = 私有 SDK 换皮

满足才值得做"开放 spec"。否则就是 NIH（Not Invented Here）的反向：发明个标准但只有自己用。
