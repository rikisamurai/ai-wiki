# Context Rot（上下文衰减）

> [!note] TL;DR
> 随着 [[context-window|Context Window]] 越来越长，模型表现会变差——注意力被分散到更多 token 上，**早期遗留的、已经无关紧要的内容开始干扰当前任务**。这不是 bug，是注意力机制的固有代价。100 万 token 的窗口比 20 万 token 更容易翻车，不是更不容易。

## 现象

典型场景：

- 长会话中突然"忘了"前面定下的约束
- 关注点漂移到几小时前讨论过的无关话题
- 之前修好的 bug 又被改坏（因为模型注意力被旧文件吸引）
- 工具调用结果越来越啰嗦，因为模型在试图覆盖更多"看到的"东西

> [!example] 一个高频翻车场景
> 让 Claude 改 A 模块，调试了半小时——窗口里塞满了 A 模块的代码、日志、报错。你再让它改 B 模块的一个小问题，它会试图把 B 模块"拉成 A 模块的风格"。原因不是它没听懂，是 A 模块占满了它的注意力分配。

## 为什么发生

注意力机制对每个新 token 都要"看一眼"窗口里的所有 token，做加权。窗口越长：

- 单个 token 拿到的注意力权重平均越小
- 噪声 token（早期对话、被废弃的尝试、读过但用不上的文件）跟有用 token 抢权重
- 模型在压缩历史时容易做出错误判断（最该保留的东西可能恰好是那个被忽略的）

这跟人脑很像：会议开了 4 小时之后，前 30 分钟讲的事你大概率记不清了。

## 它和 Compaction 翻车的因果链

> [!warning] 最危险的时刻：模型在 [[compact-vs-clear|/compact]] 那一刻智商最不在线
> 自动 compaction 触发时，窗口已经撑到极限——这正是 Context Rot 最严重的时候。让一个"已经被噪声淹没的"模型去**判断什么内容重要、什么内容可以丢**，结果可想而知：把你接下来要用的关键信息当垃圾扔了。

正确姿势是 [[compact-vs-clear|主动 /compact]]，在窗口还有充裕余量、模型还清醒时手动触发，并明确告诉它接下来要做什么、什么必须保留。

## 怎么应对

| 方法 | 何时用 |
|---|---|
| 主动 [[compact-vs-clear\|/compact]] 带指令 | 知道要换主题前 |
| 直接 [[compact-vs-clear\|/clear]] | 任务真的换了，不留也行 |
| [[rewind-胜过纠正\|/rewind]] 回到没污染的节点 | 知道哪一步开始走偏 |
| [[subagent-上下文隔离\|派 subagent]] 干脏活 | 知道某段任务会产生大量阅后即焚的中间结果 |

核心原则：**别让窗口被噪声塞满**。每次往里加东西前，问一下"这东西后面还要用吗"。
