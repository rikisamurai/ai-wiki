# 冻结快照模式（Frozen Snapshot Pattern）

> [!note] TL;DR
> Hermes（NousResearch）的发明：**会话开始时把系统提示词拍成一张"照片"封存起来，整个会话期间绝不重拍、绝不修改**。临时插入、记忆更新、上下文压缩——所有动作都绕开这张照片。结果：[[cache-命中率]]稳定接近 100%，多轮对话节省约 75% 输入 token 成本。

## 类比：旅行前的合照

旅行开始前给全家人拍一张合照然后把相机收起来——接下来不管路上谁换了件衣服，那张合照始终是出发时的定格，不会因为细节变化就重拍一遍。系统提示词就是这张合照。

## 实现要点

Hermes 的 `agent/prompt_caching.py` 里 `apply_anthropic_cache_control()` 函数把 Anthropic 的 4 个缓存断点全部用满：

| 断点 | 内容 | 缓存效果 |
|---|---|---|
| 1 | 系统提示词 | 全会话共享，命中率 ~100% |
| 2 | 倒数第 3 条非系统消息 | 滚动窗口缓存 |
| 3 | 倒数第 2 条非系统消息 | 滚动窗口缓存 |
| 4 | 最后一条非系统消息 | 为下轮预热 |

这套布局叫 **"system_and_3"** 策略：1 个断点固定钉死系统提示词，3 个断点维护滚动对话窗口。

## 关键约束：所有"想插队"的操作都得绕开

冻结快照不是"努力别动"，而是**架构上不能动**：

- 临时系统提示词（ephemeral system prompts）需要插入 → 单独通道处理，绝不动主前缀
- 中途的记忆注入（memory injection）要更新 → 走对话消息层，主前缀纹丝不动
- 上下文压缩触发 → 压缩对话记录，系统提示词那张"照片"完整保留

任何破坏快照完整性的操作都被设计层面禁止。这是 [[稳定前缀-动态后缀]] 法则的极端形态：不仅静态在前、动态在后，**静态部分在会话生命周期内任何情况下都不允许修改**。

## 与其他框架的对比

| 框架 | 策略 | 特色 |
|---|---|---|
| **OpenClaw** | cache-prefix boundary | 跨 Provider 统一 |
| **Hermes** | system_and_3 + 冻结快照 | 用满 4 断点 |
| **SWE-agent** | last_n_messages: 2 | 简洁但受 API key 配额限制 |
| **OpenHands** | 单标记 + API 自动维护 | 最简实现 |

冻结快照的代价是**前期工程投入**——需要在框架层抽象出"主前缀"和"插入点"的明确分离。回报是命中率稳定的天花板。
