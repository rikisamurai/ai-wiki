[[cursor|Cursor]] 在生产监控里给工具调用错误分了**预期错误（expected）**和**未知错误（unknown）**两大类，预期错误再细分 5 个子类。这个 taxonomy 是 [[self-healing-loop|Self-Healing Loop]] 在工具层的具体实现——**先分类才能定阈值告警**，不分直接告警会被预期错误淹没。

## Cursor 的分类

> [!example] 5 个预期错误类型
> | 类型 | 含义 |
> |---|---|
> | **InvalidArguments** | 模型生成了不合法参数（拼错路径、类型错、超出范围） |
> | **UnexpectedEnvironment** | 上下文窗口里有矛盾信息（模型以为有的文件其实没了） |
> | **ProviderError** | 工具上游服务挂了（GenerateImage、WebSearch 之类外部 API） |
> | **UserAborted** | 用户主动取消 |
> | **Timeout** | 工具执行超时 |
>
> 5 类合起来覆盖**绝大多数**正常运行中会出现的 tool error。剩下的归 **unknown**——任何 unknown 都被 Cursor 当 bug 处理。

## 为什么要把"预期"和"未知"分开

> [!important] 阈值告警的前提
> 如果不分类，所有 tool error 走同一指标，告警就难做：
> - 设阈值低 → 正常的 ProviderError 也告警，运维麻木
> - 设阈值高 → unknown 错误（真 bug）被淹
>
> 分类后：
> - **Unknown 错误率 > 0**：任何 unknown 都告警（因为 unknown = 一定是缺陷）
> - **预期错误**：用 anomaly detection——按 (tool, model) 算 baseline，显著高于 baseline 才告警
>
> 这是给自动化监控装"过滤层"，让人类只在真正异常时被叫醒。

## 按 (tool, model) 算 baseline 的细节

> [!tip] 不同模型对同一工具的预期错误率天然不同
> Claude 用 string-replace 工具的错误率和 OpenAI 模型用的不一样——单一全局 baseline 会误报。Cursor 的解：**per (tool, model) baseline**，每个组合独立基线、独立异常检测阈值。
>
> 这跟 [[per-model-harness|per-model harness]] 同源——**模型 × 工具 是真实的二维空间，平均掉一个维度都会漏信号**。

## 工具调用错误对 [[wiki/agent-engineering/context/context-rot|Context Rot]] 的放大

> [!warning] 一个 tool error 不止"少一次结果"
> Cursor 强调的关键观察：失败的工具调用会**留在上下文里**——既浪费 token 又**降低后续决策质量**（context rot 加剧）。所以工具错误率不只是工具自身的问题，还会**通过上下文污染** degrade 整个 agent 后续 turn 的表现。
>
> 这是为什么 Cursor 要花一整个 sprint 把意外工具调用错误"降低一个数量级"——回报远不止"少几次失败"，而是"少一大堆衍生失败"。

## 99%+ / 99.9% 可靠性目标

> [!example] Cursor 的硬指标
> 文章里展示的图表："**所有工具调用可靠性提升到至少 99%，很多甚至达到 99.9%**"。这个数量级跟 SRE 领域 SLO 接近——agent 工具不是"能用就行"的玩具，必须像 production service 一样订阶段性可用性目标。

## 与 [[self-healing-loop|Self-Healing Loop]] 的衔接

每周自动 agent 跑日志扫**新出现 / 近期激增** tool error，用本 taxonomy 分类后建/更新 Linear 工单。这就是 Self-Healing Loop 在 Cursor 的具体落地。任何工具错误率的异常都会触发：

```
日志扫描 → 分类（这页 taxonomy）→ 异常检测 → 工单创建/更新 → 修复 → 验证（监控曲线回归基线）
```
