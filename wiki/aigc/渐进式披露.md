[[agent-skills|Skill]] 是一个**文件夹**，不是单个 markdown。把整个文件系统当作 Context Engineering 的工具：**不要一次性把所有信息塞给模型**，而是告诉 Claude 有哪些文件、什么场景下该读，让它在需要时再去 Read。这就是渐进式披露——节省 [[context-window|上下文窗口]]，避免无关信息稀释关键信号。

> [!example] 典型目录结构
> ```
> my-skill/
>   SKILL.md            # 主入口，含 description + 何时触发
>   references/
>     api.md            # 详细函数签名（按需 Read）
>     edge-cases.md     # 踩坑点合集（按需 Read）
>   assets/
>     output-template.md  # 输出模板
>   scripts/
>     fetch-data.sh
> ```

**SKILL.md 应该是索引而非全文**——只放"什么时候用 / 怎么开始 / 哪些文件分别做什么"，把细节文档下沉到 `references/`，把模板下沉到 `assets/`。这与 [[隐性知识与上下文]] 的核心原则一致：低信噪比内容不要常驻上下文。

> [!tip] 与"存储脚本"的协同
> 同时把脚本和库放进 Skill 目录（见 [[skill-编写实践]] 的"存储脚本与生成代码"部分）——Claude 的精力花在**组合编排**而非重写样板。脚本本身不进上下文，只在被调用时执行；这本质上也是渐进式披露——把"代码逻辑"下沉到文件系统，模型只看到"工具签名"。

**反例**：把所有 API 签名、所有踩坑点、所有示例都堆在 SKILL.md 里——会出现两个问题：① 每次会话都消耗大量 token 加载用不上的内容；② 关键的"何时触发"信号被淹没。

> [!example] 同一原则在项目根目录的应用：AGENTS.md
> [[agents-md|AGENTS.md]] 也应该是目录而非百科全书。OpenAI Codex 团队的最终结构：`AGENTS.md`（~100 行入口）+ `docs/{design-docs, exec-plans, product-specs, references}/` 真正的 System of Record。SKILL.md 之于 Skill 文件夹，等价于 AGENTS.md 之于项目根目录。
