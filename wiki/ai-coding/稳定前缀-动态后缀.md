# 稳定前缀-动态后缀

> [!note] TL;DR
> Agent prompt 设计的黄金法则：**静态内容在前，动态内容在后**。绝不在静态内容之前插入任何动态信息（时间戳、用户 ID、请求号、心跳）。这条边界 OpenClaw 起名 **cache-prefix boundary**，Anthropic 内部代号 **DYNAMIC_BOUNDARY**——一边是"永远不变"，一边是"每次都在变"，越界即破坏 [[prefix-cache]]。

## 标准结构

从前往后排列：

```
[ 系统人设 ]              ← 永远不变
[ 工具定义 / Schema ]     ← 工具版本变才变
[ 长期参考文档 / RAG ]    ← 文档版本变才变
─────── cache-prefix boundary ───────
[ 对话历史（只追加） ]    ← 每轮新增
[ 当前用户输入 ]         ← 每次都新
[ 时间戳 / 请求 ID ]     ← 永远变化
```

## 为什么这条边界这么硬

[[prefix-cache|Prefix Cache]] 是**字节级精确匹配**。一旦在静态区掺入任何动态字节，从该位置起的所有 KV 缓存全部作废。20,000 tokens 的工具定义本来只算一次，因为前面多了一个时间戳，**每步都要重算**。

这就是为什么 [[cache-失效陷阱]] 的第一条永远是"别在系统提示词开头插时间戳"——它不是建议，是物理约束。

## 工具定义本身也要稳

工具定义是 Agent prompt 中最大的静态块（生产系统常达 20,000+ tokens）。要把它真正"稳"下来需要三件事：

- **打缓存标记**：在工具列表的最后一个工具上打 `cache_control`（Anthropic）
- **稳定 key 排序**：JSON 序列化用 `sort_keys=True`，否则 [[cache-失效陷阱#3.-json-序列化-key-顺序不稳定|Go map / Swift dict 的随机顺序]]会偷偷打穿你
- **不动态增删工具**：用 `tool_choice` 参数限制本次可用工具，而不是修改工具列表本身

## 案例：OpenClaw 的实际收益

执行这一条边界后（详见 [[long-horizon-agent]]）：

| 指标 | 优化前 | 优化后 |
|---|---|---|
| 每次会话 prompt token | ~800K | ~200K |
| 后续轮次延迟 | 10–16 秒 | 1–2 秒 |
| Cache 命中率 | 84% | ~95%+ |

## Anthropic 自己怎么实现这条边界

逆向 Claude Code 源码会发现 Anthropic 把系统提示词切成 4 个 block，**Block 3（静态）和 Block 4（动态）之间的分界线就是 `DYNAMIC_BOUNDARY`**：

```
Block 1: 计费归因头              不缓存
Block 2: CLI 前缀                不缓存
Block 3: 静态指令（行为规则）     ← global 缓存（全球用户共享）
─── DYNAMIC_BOUNDARY ───
Block 4: 动态内容（CLAUDE.md）   ← org 缓存
```

切分函数 `splitSysPromptPrefix()` (api.ts:321) 干的就是这件事。详见 [[prefix-cache#claude-code-的内部-4-block-结构|prefix-cache § Claude Code 4-block 结构]]。

> [!tip] 为什么 CLAUDE.md 改一次很贵
> CLAUDE.md 在 Block 4 的位置——改它会让 Block 3 的 global 缓存还在，但 Block 4 起的所有内容（含工具定义和消息历史）全部失效。所以"配好就别动"是真理。

## 进阶模式

如果会话很长，需要"系统提示词在整个会话生命周期里都不变"，参考 Hermes 的 [[冻结快照模式]]——会话开始时把系统提示词拍成"照片"封存，临时注入、记忆更新、上下文压缩都绝不动这张照片。
