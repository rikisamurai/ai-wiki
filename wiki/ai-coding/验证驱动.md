让 Agent 自己验证自己的工作，是 Claude Code 最佳实践里**最高杠杆**的做法——提供测试、截图或预期输出，Claude 就能进入"试-错-改"自我闭环；没有验证标准，它只能产出 [[plausible-code|似是而非的代码]]，剩下的检查工作全压回到你身上。

> [!important] 不是让 Claude 写测试，是给 Claude 测试
> "验证驱动"不等同于 TDD。重点是**输入端**：在你下指令的那一刻就把"怎么算成功"摆在桌面上——一组测试用例、一张设计稿、一条期望日志——而不是等 Claude 写完再问"这对不对"。

> [!compare] 提示词改造前后
> | 策略 | 改进前 | 改进后 |
> |---|---|---|
> | **提供验证标准** | "实现一个邮箱验证函数" | "写 validateEmail，测试用例 `a@b.com`→true、`invalid`→false。实现后跑测试" |
> | **视觉验证 UI** | "让仪表盘更好看" | "[贴截图] 实现这个设计，截图对比差异并修复" |
> | **定位根因** | "构建失败了" | "构建报错 [贴日志]。修复并验证构建成功，要解决根因而非压制错误" |

**验证手段的层级**

由强到弱排：① 测试套件 ② Linter / Type Check ③ Bash 命令（构建/启动）④ 截图对比 ⑤ Chrome 扩展实时检查 UI ⑥ 你肉眼 review。前面几档越能落地，[[review-带宽瓶颈|review 带宽]] 越省。

**与 [[self-healing-loop|Self-Healing Loop]] 的关系**：验证标准就是 Self-Healing Loop 的"反馈信号源"。没有验证 = Loop 失去关闭条件，Agent 跑得再快也只能输出"看起来对"的代码。这也是 [[harness-engineering|Harness Engineering]] 的核心理念之一——投资可靠的反馈机制比投资 prompt 工程回报更高。

> [!warning] "信任-验证缺口"是 Claude Code 反模式之一
> 官方文档把"产出看似合理但未处理边界情况"列为 5 个高频反模式之一。解决方案永远是同一句：**始终提供验证手段**（测试、脚本、截图）。
