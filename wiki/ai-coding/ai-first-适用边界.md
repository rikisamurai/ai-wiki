# AI First 适用边界

> [!note] TL;DR
> 即便 [[ai-first-工程前提|五件工程前提]]全做到了，[[ai-first-vs-ai-assisted|AI First]] 也只适合一部分场景。**适合**：后端为主、UI 不复杂、效果可数据化验证的产品。**不适合**：UI 密集、质量敏感、安全要求高的场景。看产品类型选模式，不要一刀切。

## 适合 AI First 的场景

> [!example] 三类适合的产品
> - **后端逻辑为主、界面不复杂的产品**：API 服务、数据处理平台、内部工具。功能好不好跑数据就知道，不需要人盯着每个像素
> - **早期产品快速试错**：功能上了不行就撤，用户预期本来就不高，AI 的速度优势能充分发挥
> - **Agent 平台 / 后端驱动产品**：CREAO 自己就是个 Agent 平台——能自己重建自己，说明这条路走得通

共同特征：**反馈可以自动化、可以量化**。AI 出错的代价是数据不好看，不是用户骂街或者赔钱。

## 不适合 AI First 的场景

### UI 密集的产品

自媒体天天喊"前端已死"，但你让 AI 做个复杂界面试试——易用性、交互细节、视觉还原，它搞不定。

> [!warning] 一个反讽
> 马斯克要是真能靠 AI 改 X 的 UI，早就改了不知道多少版了。事实是直到现在他也得靠人改。

### 功能质量敏感的产品

> Anthropic 和 OpenAI 不知道 AI First 吗？他们敢在 Claude Code 和 Codex 上这么搞吗？让 AI 全自动迭代自家的核心产品，用户不骂死才怪。

核心产品的小 bug 都会被放大到全网讨论——AI 在这种产品上的"试错+回滚"模式直接破产。

### 安全性要求高的场景

银行系统、在线交易平台、医疗系统——AI 代码出个差错，**那可不是回滚能解决的**。这类产品上 AI 充其量做辅助，主流程必须人审。

## 判断的简洁框架

| 维度 | 适合 AI First | 不适合 |
|---|---|---|
| 反馈可量化 | ✅ | ❌（要靠用户感知） |
| 错误代价可承受 | ✅（撤回功能就好） | ❌（错了就是事故） |
| UI 复杂度 | 低 | 高 |
| 用户预期 | 接受迭代 | 期待稳定 |
| 上线节奏 | 当天可撤 | 上线就要稳定数年 |

四票里有三票是"不适合"，就别勉强 AI First。

## 一个公司里可能两种模式并存

CREAO 是 Agent 平台所以全栈 AI First；但同样的公司，如果在做"给银行的核心交易系统"，那块业务就该是 AI-Assisted。

**别被"AI First 全员推行"的口号绑架。** 看每个业务的产品类型选模式才是工程决策。
