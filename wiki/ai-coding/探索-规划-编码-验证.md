Claude Code 推荐的标准工作流：**探索 → 规划 → 编码 → 验证**。让 Agent 直接跳到编码可能解错问题，分四步走能强制 Agent 先理解上下文、写计划等你审，再动手——本质是把 [[plan-mode|Plan Mode]] 的"思考-执行分离"扩展成完整流程。

> [!example] 四阶段拆解
> 1. **探索**——让 Claude 阅读相关代码、理解现有模式（只读）
> 2. **规划**——让 Claude 输出方案，**你审核**后再继续
> 3. **实现**——按已审计划编码
> 4. **验证**——跑测试 / [[验证驱动|验证手段]] 确认正确

> [!tip] 何时跳过规划
> 任务范围清晰且改动很小（修 typo、加日志、重命名变量），可以直接执行。**经验法则：如果你能一句话描述 diff，就跳过规划阶段**。

**为什么"规划"独立成阶段**：在 Claude 还没写代码时审计划，纠错成本最低；等它已经改了 8 个文件再发现方向错了，要么 [[rewind-胜过纠正|rewind]] 损失工作、要么硬改造成 [[plausible-code|似是而非的代码]]。这与 [[spec-coding|Spec Coding]] 的核心理念同构——**约束写得越早越省事**。

**与 [[harness-engineering|Harness Engineering]] 的关系**：四阶段工作流是把 [[harness-成熟度|Harness 三层成熟度]]里 L3 Workflow Automation 的标准模板"硬编码"进了 Claude Code 的产品形态——`Plan Mode` + `Edit Mode` 的双态切换是这个模板的载体。
