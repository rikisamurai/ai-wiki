# Self-Healing Loop（自愈反馈循环）

> [!note] TL;DR
> [[ai-first-vs-ai-assisted|AI First]] 体系的灵魂：每天定时由 AI 自动扫日志、聚类错误、建工单、派给合适的人，工程师修完后由同一条流水线验证、关闭工单——**整条链路以最少的人工干预完成"检测 → 分诊 → 修复 → 验证"**。CREAO 实测在过去 14 天里平均每天 3–8 次生产部署。

## 核心循环（CREAO 的具体实现）

```
[每天 09:00 UTC]
    ↓
Claude Sonnet 4.6 查询 CloudWatch
    ↓
分析所有服务的错误模式
    ↓
生成系统健康执行摘要 → 团队聊天群
    ↓
[一小时后]
    ↓
分诊引擎启动
    ↓
将生产环境错误信息聚类
    ↓
9 个维度评估每个问题严重程度
    ↓
任务管理系统自动建工单（含日志样本、受影响用户、接口、建议方向）
    ↓
自动去重 / 检测倒退（Regression）
    ↓
分配给合适的工程师
    ↓
工程师提交修复 → 走标准 6 阶段流水线
    ↓
分诊引擎重新检查监控数据
    ↓
错误解决 → 工单自动关闭
```

整个流程**没人主动吩咐**，每天自己跑。

## 为什么这是 [[ai-first-vs-ai-assisted|AI First]] 的"灵魂"而不是锦上添花

传统模式下 bug 处理的链路是：

```
监控告警 → 值班看到 → 创建 ticket → PM 排优先级 → 分配开发 → 修复 → QA → 发版
```

每一步都靠人推动，**人成为整条链路的速度上限**。AI First 模式下，AI 不只是替工程师写代码，**它替整条链路做调度**——监控发现问题、AI 分诊、AI 派单、AI 审查修复 PR、AI 验证修复效果。人只在"判断有没有战略风险"这一个节点出现。

> [!tip] 从"人催 AI 干活"到"AI 催人干活"
> 传统：工程师建 ticket → AI 帮写代码  
> Self-Healing Loop：监控触发 → AI 建 ticket、给诊断 → 派给人 → 人验证后 AI 提 PR

## 配套的"工具单一职责"原则

自愈循环背后的工程纪律是 [[worse-is-better]] 风格的：每个工具只负责一个阶段，**没有哪个工具试图包揽一切**。

| 工具 | 单一职责 |
|---|---|
| CloudWatch | 暴露结构化指标 |
| Sentry | 捕获结构化异常 |
| Claude（分诊引擎） | 错误聚类、严重度评估 |
| Linear | 面向人类的工单界面 |
| GitHub Actions | 6 阶段流水线 |
| Statsig | 功能开关 / 灰度 / 一键关闭 |
| Graphite | 合并队列 |

每个工具职责清晰，组合起来才能形成可被 AI 编排的链路。

## 必备的工程基础

Self-Healing Loop 跑得起的前提是 [[ai-first-工程前提]] 五件事**全部到位**。特别强调三条：

- **结构化日志 + 25+ 自动告警**："AI 读不懂日志，就无法诊断问题"
- **6 阶段确定性流水线**："不允许任何人工强行绿灯"——AI 才能预测结果
- **Feature Flag + 一键关闭**：糟糕的功能上线当天就死掉

## 实际效果

CREAO 实测：

- 14 天里平均每天 **3–8 次生产部署**（旧模式两周做不出 1 次）
- 糟糕功能上线当天被砍掉
- 新功能在构思当天就能上线
- 用户参与度 ↑、付费转化率 ↑（反馈闭环极短 → 学到的更多）
