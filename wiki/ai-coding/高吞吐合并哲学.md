当 Agent 吞吐量远超人类注意力时，传统工程规范变得**适得其反**——核心反转是：**纠正成本低，等待成本高**。OpenAI Codex 团队的实践：3 人扩到 7 人后日均 PR 仍在持续增长（人均 3.5 PR/天），靠的就是把"阻塞性合并门槛"压到最小。

> [!warning] 适用条件先说在前面
> 这一套**仅在高吞吐 + 强 Harness 环境下**成立。在低吞吐量环境下这样做是不负责任的。前置条件包括：完备的 [[harness-engineering|Harness]]、自动化的 [[doc-gardening|垃圾回收]]、Agent-to-Agent 的 Review 闭环。

**具体策略**

- **最小化阻塞性合并门槛**：能 follow-up 解决的问题不阻塞主干
- **PR 生命周期短**：3.5 PR/人/天意味着 PR 的中位生命周期只有几小时
- **Flaky Test 用重跑解决**：而不是无限期阻塞进度
- **Agent-to-Agent Review 替代 Human Review**：Human Review 不是必须的——随时间推移几乎所有 Review 工作交给 Agent

**为什么倒置**

- **传统假设**：每个错误都贵（要回滚、要复盘、要发邮件道歉）→ 提高合并门槛降低错误率
- **Agent 假设**：每个错误的修复极便宜（Agent 几分钟开 fix PR）→ 等待 Agent / 人类 Review 才是更大的浪费

这与 [[agent-等待时间|Agent 等待时间]] 是同一回事的两面：等待时间不止包括"Agent 等人 Review"，也包括"Agent 等 PR 合并"。把合并门槛压到最低，就是消灭这一类等待。

**和 [[review-带宽瓶颈]] 的对照**：在低吞吐人类主导环境，Review 是瓶颈，必须谨慎。在高吞吐 Agent 主导环境，Review 不再是瓶颈而是**可以工程化掉的中间环节**——靠的不是少 Review，而是把 Review 自动化、Agent 化、并行化。
