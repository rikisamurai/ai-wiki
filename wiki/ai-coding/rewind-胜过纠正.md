# Rewind 胜过纠正

> [!note] TL;DR
> 如果只能挑一个能代表"优秀上下文管理能力"的好习惯，那一定是用好 **`/rewind`**（或双击 Esc）。在纠正 AI 错误时，**回到错误发生前的节点重新下指令**，比在错误之后追一条"换 X 方法试试"更高明。前者保留了已学到的教训但抛弃了被污染的轨迹，后者把错误尝试永久焊在了上下文里。

## 操作

```
双击 Esc      ← 触发回溯
   或
/rewind       ← 等价的命令
```

让你穿越回任意一条历史消息，**该节点之后所有对话从上下文中彻底抛弃**。

## 经典场景对比

> [!compare] 同一个 bug 两种处理
> 假设 Claude 读了 5 个文件、试了一种方法 A、失败了。你的目标是让它换 B 方法。
> 
> **本能反应（追加纠正）**：
> ```
> 用户：这招不管用，换 B 方法试试
> ```
> Claude 重新尝试，但上下文里**永久保留了** A 方法的失败痕迹（5 个文件的内容、A 方法的尝试代码、报错信息）。后面每一步都被这堆噪声拖累。
> 
> **更高明的做法（rewind）**：
> 1. rewind 回到"它刚读完那 5 个文件"的节点
> 2. 重新发指令：
> ```
> 用户：别用 A 方法了，foo 模块根本不支持那个——直接去试 B 方法
> ```
> Claude 带着教训重新出发，上下文干净。

## 为什么这是上下文管理的核心技巧

每次"追加纠正"都在往窗口里**塞更多噪声**：

- 失败的尝试代码（再也用不上）
- 错误日志（已经知道结论了）
- 模型解释为什么失败（不再相关）
- 你纠正它的对话（meta，模型可能反而被它误导）

这些噪声会触发 [[context-rot|Context Rot]]，并污染 [[prefix-cache|Prefix Cache]] 命中——因为前缀变了。

> [!tip] Rewind 是反熵操作
> 大部分会话动作都是熵增（往窗口加东西）。Rewind 是少有的**熵减**动作——它真的把噪声从上下文里删除。这就是它特别有价值的原因。

## "从这里开始总结"——rewind 的进阶玩法

`summarize from here` 让 Claude 自己把"刚学到的教训"压成一段交接说明：

> 这感觉就像那个刚刚踩了坑的"未来版 Claude"，给过去那个还没开始行动的自己留下了一张字条。

具体玩法：

1. 让失败的尝试跑完，看到完整教训
2. 让 Claude `summarize from here`：抽出关键教训、避坑要点、推荐路径
3. Rewind 回到原始任务起点
4. 把那张"字条"贴进去，重新启动

得到的是"带着先验知识的干净上下文"——比单纯 rewind 更强。

## 什么时候 *不* 适合 rewind

- 教训本身需要保留并展示给后续的人/AI 看（比如生成 PR 描述时要解释为什么没走 A 方案）
- 失败尝试中产生的代码片段还想用一部分
- Rewind 会丢掉的中间状态（比如已经创建的文件）需要手动清理

这些场景下用 [[compact-vs-clear|/compact 带指令]] 或 [[会话管理动作|让 subagent 做]]更合适。
