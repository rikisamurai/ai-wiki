大多数 [[harness-engineering|Harness]] 体系擅长捕捉**结构性错误**（代码能否运行、测试能否通过），但存在系统性盲区：**行为正确性**——代码是否真的做了用户期望的事。一个"全绿"的 PR 完全可能实现了错的需求。

> [!warning] 三类常缺的验证
> - **用户价值 E2E 验证**：核心用户路径是否符合真实使用场景？
> - **意图失败检测（Intent Failure）**：代码通过了所有测试，但实现的是错误的需求？
> - **对抗性数据集**：边界输入能否暴露隐藏的逻辑错误？

这不是 Harness 的替代方案，而是 Harness 需要覆盖的**下一层**。如果你目前的 Harness 只有单元测试和 Lint，行为正确性验证就是下一个建设目标。

**和 [[plausible-code]] 是一对孪生概念**：plausible code 关注"代码看起来对但实际不对"，行为正确性关注"测试都过了但需求实现错了"。前者在代码层、后者在意图层，两者都是"绿色信号≠真信号"的不同表现。

在 [[harness-成熟度|Harness 三层成熟度]]里，行为正确性属于 **Entropy Management** 层——对抗 AI 长期产出 slop 的核心能力，也是最容易被忽视的一层。
