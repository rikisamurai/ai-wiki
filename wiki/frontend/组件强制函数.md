"强制函数"指一条**约束规则被设计出来不是为了规则本身，而是为了倒逼上游设计变得更清晰**。Factory 团队的 [[wiki/frontend/no-useeffect-rule|No useEffect 规则]] 就是这种机制——它表面上禁的是一个 Hook，实际倒逼出来的是更干净的组件树。

> [!note] 强制函数的特征
> - 看起来像"限制"，实质是"设计指南"
> - 不告诉你"应该怎么做"，而是关掉"最容易选错的那条路"
> - 短期不舒服，长期把团队代码风格收敛到一个更高的基线

## 在 React 组件设计中的体现

禁用裸 useEffect 之后，组件树自然分化为三种角色：

| 角色 | 职责 | 工具 |
| --- | --- | --- |
| **父组件** | 拥有编排和生命周期边界 | 条件渲染、[[wiki/frontend/key-重置组件\|key]] |
| **子组件** | 假设前置条件已满足，专注 UI | [[wiki/frontend/派生状态\|派生状态]]、handler |
| **Effect 边界** | 与外部世界同步（DOM、API、订阅） | [[wiki/frontend/usemounteffect\|useMountEffect]] |

这基本上是 **Unix 哲学应用于 React 组件**：每个单元做一件事，协调发生在清晰的边界。

**为什么直接说"写干净组件"不行**：抽象规则（"写干净的组件"）每个开发者理解都不一样，AI Agent 更没法判断什么算干净。强制函数的好处是**可机械验证**：

- lint 规则能扫出违规
- AGENTS.md 能告诉 AI 该用什么替代
- code review 不需要主观争论

## 类似的强制函数模式

强制函数不是 React 独有，是个通用工程范式：

- **Rust 的 borrow checker**：不让你写不安全的代码，倒逼你想清楚所有权
- **Erlang 的"let it crash"**：禁用 try/catch 写法，倒逼你把容错放到 supervisor 层
- **Go 的"无 generics 时代"**：限制让 API 设计被迫扁平
- **Tailwind 的"无自定义 CSS"**：约束设计 token，倒逼 design system 收敛

**在 AI 编码时代尤其重要**：[[wiki/ai-coding/agentic-coding|Agentic Coding]] 的失败模式很多源自 Agent "选了一条似是而非的路径"。强制函数把"最容易选错的路"封死，比让 Agent 自己判断风险更可靠。这跟 [[wiki/ai-coding/plausible-code|似是而非代码]] 的防御逻辑是一致的。
